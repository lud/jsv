Preemptive robustness/security hardening pass over JSV, prompted by the wave of
vulnerabilities reported across schema/parsing libraries. This groups the
findings from an audit run against the "untrusted schema + untrusted data"
threat model.

## Framing

In normal usage schemas are trusted: they live in the codebase or in a
first-party repository, and developers are expected to treat externally-supplied
schemas with care. So none of these are considered urgent, and **none warrant a
published CVE**. They are quality/robustness improvements that make the library
safer to use against dynamically-built or untrusted schemas, and better-behaved
under adversarial data.

Each spec is scoped so it can be discussed and implemented in its own session.

## Specs (by priority)

1. `002.spec` — Validation recursion guard for `$ref`/`$dynamicRef` no-progress
   cycles. The only finding confirmed to hang the VM; highest value.
2. `004.spec` — Harden the HTTP resolver (redirects/SSRF, TLS verification,
   allow-list matching, disk cache, size/timeout).
3. `005.spec` — Atom-creation safety; flip the `:atoms` build default (already
   earmarked `# TODO(v2)`).
4. `006.spec` — Regex match limits for `pattern`/`patternProperties`.
5. `003.spec` — Graceful build errors for malformed schemas (`$dynamicAnchor`
   without `$id` crash).
6. `007.spec` — Gate the `jsv:module:` internal resolver (low).

## Verified clean during the audit (no work needed)

Builder `$ref` cycles (memoized), deep schema build and deep data validation
(BEAM heap-grown stack), `uniqueItems` (map-keyed, O(n)), `multipleOf`/Decimal
(guarded + rescued, `multipleOf: 0` rejected at build), `format: "regex"`
(compile-only, PCRE2 self-guards), the normalizer (atom→string only), and the
`x-jsv-cast` chain (module via `to_existing_atom`; the target function is chosen
by the trusted module's `__jsv__/2`, not by the schema). No `eval`,
`binary_to_term`, `Module.concat`, or dynamic `apply` anywhere in `lib/`.
