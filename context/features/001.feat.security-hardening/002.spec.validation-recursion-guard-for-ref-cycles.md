## Problem

At validation time there is no cycle detection and no recursion-depth limit. A
schema whose `$ref` (or `$dynamicRef`) re-enters the same schema node against the
**same data pointer** recurses forever, allocating on each hop, until the process
OOMs / the BEAM dies. This is the only finding confirmed to actually hang.

Confirmed by execution (OTP 29):

```elixir
schema = %{"$defs" => %{"a" => %{"allOf" => [%{"$ref" => "#/$defs/a"}]}},
           "$ref" => "#/$defs/a"}
{:ok, root} = JSV.build(schema, atoms: false)
JSV.validate(123, root)   # never returns; ~2.6 GB in ~6s before OOM
```

Same class via dynamic refs:

```elixir
%{"$id" => "http://ex/root", "$dynamicAnchor" => "node",
  "allOf" => [%{"$dynamicRef" => "#node"}]}
```

The recursion is driven by keywords that re-validate the same data without
descending into it (`allOf`/`anyOf`/`oneOf`/`$ref`), so each hop adds stack
frames with pending continuations (`merge_tracked`/`merge_errors` run after the
recursive return) — it is not tail-recursive.

Legitimate recursive schemas over finite data must keep working. Verified that
`properties.child → {"$ref": "#"}` against nested data returns correctly, because
there the data pointer advances each hop. Any fix must key on *progress*, not
merely on ref identity.

## Where

- `lib/jsv/validator.ex`

    → `checkout_ref/2` (~521) → `validate/3` (~87)
  - `checkout_dynamic_ref/3` (~525) for the `$dynamicRef`/`$dynamicAnchor` path
- Ref keyword call site: `lib/jsv/vocabulary/v202012/core.ex:143`
- `JSV.Validator.ValidationContext` (~19) — where guard state would live

## Decided design (session 2026-07-05)

**Spec basis.** JSON Schema 2020-12 core §9.4.1 "Guarding Against Infinite
Recursion": implementations must terminate ("A schema MUST NOT be run into an
infinite loop against an instance"), while recursive nesting in schemas is only
discouraged ("Schemas SHOULD NOT [...]; the behavior is undefined"). Such
schemas are therefore buildable, and any terminating outcome is conformant.

**Decisions:**

1. **Progress-keyed seen-set in the validator** (no build-time detection: it
   cannot be complete — `$dynamicRef` resolution depends on the runtime dynamic
   scope, and static graph cycles through conditionals are not guaranteed
   runtime loops — and once the runtime guard exists it is redundant). No
   `max_depth` option.
2. **Silently pass on re-entry.** A purely self-referential ref states no
   constraint; the open-world reading (empty schema passes) makes
   `a = allOf [a]` accept everything. Erroring would make it behave like
   `false`, which is more surprising.
3. **Entries are `{ref_key, data_term}` pairs** on the *active* stack, pushed in
   `validate_ref/4` and explicitly restored (popped) on return. Keying on the
   data term (not `data_path`) matters:
   - `propertyNames` validates key strings without advancing `data_path`; a
     path-keyed set would wrongly cut finite schemas like
     `a = {"minLength": 3, "propertyNames": {"$ref": "#/$defs/a"}}`.
   - A strict subterm can never equal its ancestor (finite terms), so descent
     via `validate_in/5` naturally invalidates entries; the set is still reset
     there to stay small.
   - Soundness: same `{ref, data}` on the stack ⇒ deterministic repeat ⇒ true
     loop. Completeness: an infinite run stops descending eventually (finite
     data), so some `{ref, data}` pair repeats by pigeonhole.
   - `$dynamicRef` keys on the *given* `{:dynamic_anchor, ns, anchor}` key:
     resolution favors the outermost scope, and outer scopes are stable under
     deepening, so a repeated given key resolves identically — no need to
     thread the resolved key out of `checkout_ref`.
4. **Stack semantics, not visited-set.** The suite's
   `infinite-loop-detection.json` ("evaluating the same schema location against
   the same data location twice is not a sign of an infinite loop") requires
   that sibling evaluations of the same ref be unaffected; push/pop in
   `validate_ref` guarantees this.

## Acceptance criteria

- The two PoC schemas above return promptly (valid — pass on re-entry) instead
  of hanging.
- Legitimate recursive schemas over finite data still validate correctly.
- In multi-node loops (A→B→C→A), constraints attached at *every* level are
  still executed exactly once — only the re-entering ref hop is cut.
- Multi-cycle graphs (A→B, B→C, B→D, C→A, D→B) terminate and execute each
  node's constraints.
- The `propertyNames` self-ref case above still applies `minLength` to keys.
- The JSON Schema Test Suite still passes (esp. recursive-ref and
  infinite-loop-detection cases).

## Verify

Add regression tests wrapping the PoCs in a `Task` with a timeout so a
regression cannot re-hang the suite. Cover: both PoCs, constraints at all
levels of an A→B→C→A loop (failing datum per level), the multi-cycle graph
above, the `propertyNames` case, and sibling refs at the same data level
(`allOf: [$ref X, $ref X]`).
