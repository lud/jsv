## Problem

At validation time there is no cycle detection and no recursion-depth limit. A
schema whose `$ref` (or `$dynamicRef`) re-enters the same schema node against the
**same data pointer** recurses forever, allocating on each hop, until the process
OOMs / the BEAM dies. This is the only finding confirmed to actually hang.

Confirmed by execution (OTP 29):

```elixir
schema = %{"$defs" => %{"a" => %{"allOf" => [%{"$ref" => "#/$defs/a"}]}},
           "$ref" => "#/$defs/a"}
{:ok, root} = JSV.build(schema, atoms: false)
JSV.validate(123, root)   # never returns; ~2.6 GB in ~6s before OOM
```

Same class via dynamic refs:

```elixir
%{"$id" => "http://ex/root", "$dynamicAnchor" => "node",
  "allOf" => [%{"$dynamicRef" => "#node"}]}
```

The recursion is driven by keywords that re-validate the same data without
descending into it (`allOf`/`anyOf`/`oneOf`/`$ref`), so each hop adds stack
frames with pending continuations (`merge_tracked`/`merge_errors` run after the
recursive return) — it is not tail-recursive.

Legitimate recursive schemas over finite data must keep working. Verified that
`properties.child → {"$ref": "#"}` against nested data returns correctly, because
there the data pointer advances each hop. Any fix must key on *progress*, not
merely on ref identity.

## Where

- `lib/jsv/validator.ex`
  - `validate_ref/4` (~361) → `with_scope/4` (~381) → `do_validate_ref/3` (~367)
    → `checkout_ref/2` (~521) → `validate/3` (~87)
  - `checkout_dynamic_ref/3` (~525) for the `$dynamicRef`/`$dynamicAnchor` path
- Ref keyword call site: `lib/jsv/vocabulary/v202012/core.ex:143`
- `JSV.Validator.ValidationContext` (~19) — where guard state would live

## Proposed approach (to discuss)

Two options; the first is preferred because it does not penalise legitimate deep
recursion:

1. **Progress-keyed seen-set.** Track the set of `(ref_or_anchor_key, data_path)`
   pairs currently on the validation stack in `ValidationContext`. On re-entry of
   the same pair, stop (error, or treat as non-matching) instead of recursing.
   Only true no-progress loops are cut; recursion that advances the data pointer
   changes `data_path` and is unaffected. Must cover both `do_validate_ref` and
   the dynamic-anchor resolution.
2. **Configurable max validation depth.** A simple counter with a limit exposed
   on `JSV.validate/3` (e.g. `max_depth:`). Blunt but trivial and predictable;
   would also bound genuinely-deep-but-finite data. Could sit on top of (1) as a
   backstop.

Open question for the session: when a cycle is detected, is it a validation
error, or a build-time rejection where statically detectable?

## Acceptance criteria

- The two PoC schemas above return promptly (error) instead of hanging.
- Legitimate recursive schemas over finite data still validate correctly.
- The JSON Schema Test Suite still passes (esp. the recursive-ref cases).
- If a depth option is added, it is documented on `JSV.validate/3`.

## Verify

Add regression tests wrapping the PoCs in a `Task` with a timeout so a
regression cannot re-hang the suite.
