## Problem

Schema-supplied `pattern` / `patternProperties` regexes are compiled and run
against data with `Regex.match?/2`, which passes no `:re` match limits. Erlang's
PCRE2 does its usual backtracking.

A single pathological match is **bounded** — PCRE2's built-in `match_limit` caps
a runaway and returns `false` after a while. Verified against this code: `^(a+)+$`
over `"a"×N <> "!"` costs a steady ~200 ms regardless of N (tested N = 28 … 10000,
all ~200 ms). So plain `pattern` is only a mild per-call CPU cost (LOW).

The real concern is **amplification in `patternProperties`**: the match runs as
`patterns × data keys`, each ~200 ms. Verified end-to-end: 5 patterns × 5 keys ≈
2.6 s CPU from a few-KB schema+payload; 50 × 50 (still a few KB) ≈ minutes. That
is a practical CPU-exhaustion vector under concurrency (MEDIUM).

## Where

- Compile: `lib/jsv/vocabulary/v202012/validation.ex:90-98`;
  `lib/jsv/vocabulary/v202012/applicator.ex:45-55`
- Match: `validation.ex:368-374` (`Regex.match?(re, data)`);
  `applicator.ex:302-308` (nested comprehension `patterns × keys`)
- Same shape in the draft-7 vocabulary (`lib/jsv/vocabulary/v7/...`).

## Approach

Replace `Regex.match?/2` on these hot paths with
`:re.run(re, subject, [{:match_limit, N}, {:match_limit_recursion, N}])` at a low
`N` (e.g. 1_000–10_000), treating the limit-exceeded return as "no match" (or a
distinct validation error). This drops the worst case from ~200 ms to sub-ms.
Compiled regexes are already stored per keyword, so only the run site changes.

Optional extras (discuss): a per-validation wall-clock budget as a backstop, and
documenting a recommended cap on data string length / property counts.

Note: `format: "regex"` (`lib/jsv/format_validator/default.ex:106-107`) only
**compiles** attacker data and never runs it — PCRE2 rejects pathological
patterns fast (verified), so no change needed there.

## Acceptance criteria

- A `patternProperties` schema + data crafted for backtracking completes in
  well under the current ~200 ms/match (target sub-ms per match) and still
  reports the correct validation result.
- The `match_limit` value is a module attribute / documented constant.
- The JSON Schema Test Suite pattern cases still pass (no legitimate match is
  lost at the chosen `N`).
