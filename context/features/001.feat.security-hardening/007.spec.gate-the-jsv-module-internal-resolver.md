## Problem

A schema-controlled `$ref` of the form `jsv:module:Some.Module` is resolved by
the internal resolver into a module via `String.to_existing_atom` +
`Code.ensure_compiled!`, then `JSV.Schema.from_module/1` is called, i.e.
`module.json_schema()`. So an untrusted schema can trigger a call to the
fixed-name zero-arg function `json_schema/0` on **any already-loaded module** in
the system.

## Where

- `lib/jsv/resolver/internal.ex:26-33`
- Sink: `lib/jsv/schema.ex:758-760` (`from_module/1` → `module.json_schema()`)

## Assessment

Low / informational. The surface is tightly constrained:

- Only existing atoms resolve (`String.to_existing_atom` raises otherwise → the
  `:31-33` rescue returns `{:error, {:invalid_schema_module, e}}`), so no atom
  creation.
- The invoked function name is fixed (`json_schema/0`) and takes no schema-derived
  arguments — a JSV convention rarely present on non-schema modules.
- Any raise is caught and returned as a build error.

Exploitation would need a loaded "gadget" module that exports `json_schema/0`
with a meaningful side effect — unlikely. There is no arbitrary function name or
argument control. Not a code-execution hole.

## Approach (optional)

If `jsv:module:` should never be reachable from untrusted schema input, gate it:
either behind an explicit opt-in on the resolver chain, or validate the target
against an allow-list of registered schema modules. Otherwise, document that
`jsv:module:` refs are a trusted-schema feature and leave as-is.

## Acceptance criteria

Decision recorded (gate vs document). If gated, an untrusted schema's
`jsv:module:` ref no longer resolves/invokes without the opt-in, and existing
first-party usage keeps working.

## Priority

Low.
