## Problem

The only unbounded atom-creation paths in the library are in the cast feature,
and they are gated behind the `:atoms` build option — but that option **defaults
to `true`** for backwards compatibility, with only a runtime warning. So
`JSV.build(schema)` on a schema that uses the atom casts, without an explicit
`atoms: false`, can exhaust the (never-GC'd) atom table and crash the VM.

The code already anticipates this: `lib/jsv/cast.ex:156` carries
`# TODO(v2): Default building atoms to false`.

Unbounded paths (both gated by `check_atoms_opt/1`, `cast.ex:159`):

- `string_to_atom` / `string_to_atom_or_nil` (`cast.ex:62-76`) — convert
  validated **data** to atoms at validation time. A schema with
  `{"type":"string","x-jsv-cast":["string_to_atom"]}` turns every distinct data
  string into a permanent atom.
- `aprops` / `atom_mapping` (`cast.ex:118-146`) — convert schema **property
  names** to atoms at build time; a huge `properties` map exhausts atoms at build.

Safe and unchanged: `string_to_existing_atom` (`cast.ex:56-60`), the
`string_enum_to_atom*` helpers (they also emit an `enum`, and the cast
short-circuits on `Validator.error?`), and module resolution via
`safe_string_to_existing_module` (existing-atom only). See also `007.spec` for
the module-resolution surface.

## Approach (to discuss — this is a breaking change)

- Flip `@default_atoms_build` to `false` (the `v2` plan), so untrusted/dynamic
  schemas are safe by default and `atoms: true` becomes an explicit opt-in.
- Decide the deprecation path: this changes behaviour for anyone relying on the
  implicit default. Options: do it in the next major (v2) as already planned, or
  keep the warning for one more minor and flip in v2. The warning text
  (`cast.ex:148-154`) already tells users to pass the option explicitly.
- Ensure the docs (`lib/jsv.ex` build options, cast-functions guide) state that
  `atoms: false` is required for untrusted schemas and describe the DoS.

## Acceptance criteria

- With `atoms: false` (or the new default), a schema using `string_to_atom` /
  `aprops` creates no atoms from schema or data (cast is dropped as `:nocast`).
- The behaviour change and migration are documented and land in a major bump.
- Existing tests updated for the new default; the JSON Schema Test Suite still
  passes.

## Priority

High as a footgun, but gated and already scheduled for v2 — mostly a
default-flip + docs decision.
