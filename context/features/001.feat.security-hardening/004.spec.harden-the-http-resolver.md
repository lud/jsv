## Problem

`JSV.Resolver.Httpc` fetches remote schemas with fully default `:httpc` options
(`http_get/1`, `lib/jsv/resolver/httpc.ex:100-115`: `headers = []`,
`http_options = []`). That leaves several sharp edges. The resolver already
requires an `allowed_prefixes` allow-list and is opt-in, so this is hardening of
an opt-in feature rather than a default-exposed hole — but the allow-list gives a
false sense of containment because the actual fetch is governed by `:httpc`
defaults.

This spec groups all the resolver findings; they share a file and are best done
together. Consider one task per item if the session wants finer tracking.

## Findings and fixes

### 1. Redirects bypass the allow-list (SSRF) + cache poisoning — HIGH
`autoredirect` defaults to `true`, and `check_allowed/2` (`:68-74`) only checks
the **original** URL. An allowed (or compromised, or open-redirect) host can
return `302 Location: http://169.254.169.254/…` (cloud metadata, internal admin,
etc.) and `:httpc` follows it; the internal body is returned and then written to
the disk cache **keyed by the allowed URL** (`fetch_disk_or_http/2` →
`disk_cache.({:write!, json})`), so it persists with no expiration.
Fix: pass `autoredirect: false`. If redirects must be supported, re-run
`check_allowed/2` on each `Location` (and block private/link-local/loopback IPs)
before following.

### 2. No TLS certificate verification (MITM) — HIGH
No `ssl:` options are passed, so `:httpc` does not verify the server certificate
or hostname on the supported Elixir range (`~> 1.15`, older OTP). A MITM can serve
arbitrary schema content.
Fix: pass
`ssl: [verify: :verify_peer, cacerts: :public_key.cacerts_get(),
customize_hostname_check: [match_fun: :public_key.pkix_verify_hostname_match_fun(:https)], depth: 3]`
and fail closed on verification errors. (`:public_key` is already in
`extra_applications`.)

### 3. Weak allow-list matching — MEDIUM
`check_allowed/2` uses `String.starts_with?(url, prefix)`. The docs recommend a
trailing `/` but do not enforce it, so `allowed_prefixes: ["https://example.com"]`
(no slash) admits `https://example.com.evil.com/…` and
`https://example.com@evil.com/…`.
Fix: parse both prefix and URL with `URI`, compare scheme + host (+ port) exactly,
then check the path is under the allowed path prefix. Reject configured prefixes
that have no path component (fail closed). The `http(s)://`-only scheme gate in
`resolve/2` (`:44-49`) is good and stays.

### 4. Disk cache hardening — MEDIUM (local, multi-user host)
Default cache lives in world-shared `System.tmp_dir!()` at a fully predictable
path (`url_to_cache_path/2`, `:145-151`), written with `File.write!` (follows
symlinks), no integrity check, no expiration. On a shared host this allows
symlink-clobber and cache pre-poisoning.
Fix: create the cache dir `0700`, write to a random temp name then atomic
exclusive rename (`[:write, :exclusive]` / `O_EXCL`), do not follow symlinks, and
prefer a per-user cache location over shared `/tmp`.
Note: verified there is **no path-traversal** exploit — `slugify/1` (`:153-163`)
neutralizes `/` and `..` — so this is defence-in-depth, not a hole.

### 5. Unbounded download size + no timeout — LOW/MEDIUM
`body_format: :binary` reads the whole response into memory; no size cap and no
request timeout are set. A malicious allowed host (or a redirect target from #1)
can return an arbitrarily large body.
Fix: enforce a max body size (check `Content-Length` and/or stream and abort) and
set a request/connect timeout in `http_options`.

## Acceptance criteria

- Redirects are not followed (or are re-checked against the allow-list).
- HTTPS fetches verify the certificate; an invalid cert fails the fetch.
- Allow-list matching is authority-based; the `.evil.com`/`@evil.com` bypasses
  are rejected even without a trailing slash.
- Cache writes are atomic, non-symlink-following, and in a private-mode dir.
- A response-body size cap and a request timeout are enforced.
- Existing resolver tests still pass; add tests for redirect-refusal and
  allow-list bypass rejection (mock `:httpc`).
