## Problem

A malformed but syntactically-parseable schema can crash `JSV.build/2` with an
unexpected `FunctionClauseError` instead of a clean `JSV.BuildError`. Since
untrusted (or just hand-written and wrong) schemas are in scope, the builder
should fail gracefully with a structured error for every malformed input.

Confirmed case ŌĆö a `$dynamicAnchor` without an enclosing `$id`:

```elixir
JSV.build!(%{
  "$defs" => %{"a" => %{"$dynamicAnchor" => "node",
                        "allOf" => [%{"$dynamicRef" => "#node"}]}},
  "$ref" => "#/$defs/a"
})
# ** (FunctionClauseError) no function clause matching in
#    JSV.Vocabulary.V202012.Core.find_local_dynamic_anchor/2
```

`find_local_dynamic_anchor/2` (`lib/jsv/vocabulary/v202012/core.ex:39`) only
matches `%{"$id" => _}` and has no fallback clause.

## Where

- `lib/jsv/vocabulary/v202012/core.ex:39` (`find_local_dynamic_anchor/2`), called
  from the `$dynamicAnchor`/`$dynamicRef` handling around `:39`.
- Worth a quick sweep for sibling clauses in the Core vocabulary that assume a
  well-formed schema shape and would raise rather than return a build error.

## Approach

Add a fallback clause that returns/raises a `JSV.BuildError` (via
`Builder.fail/3`) describing the malformed schema, rather than letting the
`FunctionClauseError` escape. Keep the message specific enough to point at the
offending keyword.

## Acceptance criteria

- The PoC above yields a `JSV.BuildError` (or `{:error, %JSV.BuildError{}}` from
  `JSV.build/2`), not a `FunctionClauseError`.
- A regression test covers it.
- No change to valid-schema behaviour; JSON Schema Test Suite still passes.

## Priority

Low ŌĆö robustness, not exploitable beyond a crash on a bad schema.

## Outcome (2026-07-06)

The premise of the PoC was wrong: a root schema resource is not required to
declare an `$id`, so the PoC schema is **valid** JSON Schema 2020-12. Confirmed
by adding an `$id` to the same schema, which built and validated fine. Decision
(with user): fix it as a correctness bug ŌĆö the schema now builds and
`$dynamicRef` resolves per spec ŌĆö instead of returning a `BuildError`.

Changes:

- `lib/jsv/vocabulary/v202012/core.ex` ŌĆö `find_local_dynamic_anchor/2` accepts
  id-less resources (the root) and boolean schema resources.
- `lib/jsv/vocabulary/v7/core.ex` ŌĆö draft 7 delegated unknown keywords to the
  2020-12 fallback, so `$dynamicRef`/`$dynamicAnchor` were parsed (and crashed)
  there too. They are now ignored like any unknown draft-7 keyword.

The sweep found more crash paths, all now returning `JSV.BuildError`:

- Non-string `$ref`/`$dynamicRef` ŌåÆ was `FunctionClauseError` in `URI.parse/1`.
  Fixed in `JSV.Ref.do_parse/3` (`{:error, {:invalid_ref, url}}`).
- Non-string `$anchor`/`$dynamicAnchor` ŌåÆ was `FunctionClauseError` in
  `JSV.Key.for_anchor/2`. Fixed in the resolver scan (`validate_anchor/2` in
  `scan_schema`/`scan_subschema`).
- Non-string nested `$id` ŌåÆ was `FunctionClauseError` in `split_id/1`. Fallback
  clause returns `{:error, {:invalid_id, id}}`.
- Non-string root `$id` ŌåÆ was a raw `ArgumentError`; now raises `BuildError`.
- `$ref` pointing at a non-schema value ŌåÆ was `FunctionClauseError` in
  `Builder.do_build_sub/3`. Fixed at pointer resolution
  (`check_pointed_schema/2` in `JSV.Resolver.fetch_pointer/2`).

Tests: `test/jsv/dynamic_anchor_test.exs` (dynamic anchor behaviour, including
draft 7) and `test/jsv/malformed_schema_test.exs` (structured errors with
messages pointing at the offending keyword). Full suite, credo and dialyzer
pass.
