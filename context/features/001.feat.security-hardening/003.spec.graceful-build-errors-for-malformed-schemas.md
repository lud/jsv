## Problem

A malformed but syntactically-parseable schema can crash `JSV.build/2` with an
unexpected `FunctionClauseError` instead of a clean `JSV.BuildError`. Since
untrusted (or just hand-written and wrong) schemas are in scope, the builder
should fail gracefully with a structured error for every malformed input.

Confirmed case — a `$dynamicAnchor` without an enclosing `$id`:

```elixir
JSV.build!(%{
  "$defs" => %{"a" => %{"$dynamicAnchor" => "node",
                        "allOf" => [%{"$dynamicRef" => "#node"}]}},
  "$ref" => "#/$defs/a"
})
# ** (FunctionClauseError) no function clause matching in
#    JSV.Vocabulary.V202012.Core.find_local_dynamic_anchor/2
```

`find_local_dynamic_anchor/2` (`lib/jsv/vocabulary/v202012/core.ex:39`) only
matches `%{"$id" => _}` and has no fallback clause.

## Where

- `lib/jsv/vocabulary/v202012/core.ex:39` (`find_local_dynamic_anchor/2`), called
  from the `$dynamicAnchor`/`$dynamicRef` handling around `:39`.
- Worth a quick sweep for sibling clauses in the Core vocabulary that assume a
  well-formed schema shape and would raise rather than return a build error.

## Approach

Add a fallback clause that returns/raises a `JSV.BuildError` (via
`Builder.fail/3`) describing the malformed schema, rather than letting the
`FunctionClauseError` escape. Keep the message specific enough to point at the
offending keyword.

## Acceptance criteria

- The PoC above yields a `JSV.BuildError` (or `{:error, %JSV.BuildError{}}` from
  `JSV.build/2`), not a `FunctionClauseError`.
- A regression test covers it.
- No change to valid-schema behaviour; JSON Schema Test Suite still passes.

## Priority

Low — robustness, not exploitable beyond a crash on a bad schema.
